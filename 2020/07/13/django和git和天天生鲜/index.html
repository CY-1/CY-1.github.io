    静态文件

    • 网页使用css文件 js文件 图片叫做静态文件 要存在static目录下 且static要去setting下设置
    • 在setting下STATIC_URL=”/static/“ STATICFILES_DIRS=[os.path.join(BASE_DIR,”static”)]
    • 如果修改了STATIC_URL的值 需要修改html里面对应的目录 但是本身static目录名字不变
    • 如果想在html动态的加载img图片

      中间件

    • 中间件：django给我们预留的函数接口 让我们可以干预请求和应答的请求
    • 获取游览器的IP地址
      • request.META 可以获取游览器的IP地址
    • 使用方法
      • 在自己的应用下创建middleware文件夹
      • 在文件夹下创建一个类 写上process_view(self,request, view_func, *view_args, **view__kwargs):函数所以请求都会经过这个函数 如果给这个函数返回值 也就不再会调用本来继续调用的view_func函数 （可以利用此来屏蔽一些IP）
      • 在setting下设置中间件
      • 如果有两个中间件设置了process_exception 那么调用顺序和实际使用顺序相反，先声名 后调用

        自定义模型管理类

    • admin.py中的list_display 不仅可以写字段名 也可以写方法名（也要有引号）
    • title.admin_order_field 设置排序的字段 title_short_description设置方法在管理页面上字段名 在modes.xxxField(verbos_name=”xx”)是对字段起标题


      在django_migrations中会记录迁移信息

    • 如果想重新生成迁移文件和迁移 要删除django_migrations对应信息 和migration下的对应信息

    • 后台管理页面上传

      GIT

      创建和回退

    • git init 创建版本库
    • 使用如下两条命令可以创建一个版本：
      • git add code.txt
      • git commit –m ‘版本1’
    • 使用如下命令可以查看版本记录：
      • git log
    • 现在若想回到某一个版本，可以使用如下命令：
      • git reset –hard HEAD^ 表示版本回到前一个版本
      • git reset –hard 版本号 可以回到指定的版本
    • 在git中不是完整记录版本 是记录在之前的一个版本做了什么 是依赖于前一个版本
    • 用HEAD指正指向当前版本 HEAD^ 前一个版本 HEAD^^表示前两个 HEAD1 表示前一个 HEAD100表示前100
    • git reflog 查看之前操作的记录 可以看到版本号

      工作区和暂存区

    • 工作区：git init的目录就是工作区
    • 版本库：也就是在.git 隐藏目录下的
    • 暂存区：git add 也就是添加到暂存区 git commit也就是把暂存区的修改提交到版本库
    • git status 查看目录情况
    • git commit 只会用暂存区进行修改 如果直接commit 不add 那么暂存区不会变化
    • git checkou – 文件名 撤销修改（还没加入到暂存区 更没加入版本库）
    • git reset HEAD 文件名 取消暂存

      对比文件不同

    • git diff – HEAD 文件名 和当前版本比较文件差别
    • git diff HEAD HEAD^ – code.txt 比较现在版本和前一个版本的code.txt的区别
    • 删除文件记得git add和git commit -m “描述”
    • 删除文件也可以 git checkout – 被删除的文件名 恢复文件
    • 如果没有跟踪文件 那就无法通过git找回文件

      分支

    • git 会把我们每次提交的版本串成一条时间线 这条时间线就是一个分支
    • 默认分支是主分支(master)
    • HEAD严格来说是指向master 然后master指向当前版本
    • git创建分支 是增加一个指针
    • 分支合并也就是移动master的指针
    • git branch 查看当前分支
    • git checkout -b 分支名 创建且切换到分支
    • git checkout 分支名 切换分支
    • git merge 分支名 合并分支 （如果能使用快速合并 会默认只用快速合并 也就是时间线向前移动）
    • git branch -d 分支名 删除分支

      解决冲突

    • 手动解决冲突后 add 和commit即可
    • 在一些情况没有冲突但是无法快速合并 就会合并然后做一次新的提交
      • 合并后会出现一个弹窗 要输入信息 然后ctrl+x 然后Y 然后回车
    • 快速合并没有git log记录
    • git merge –no-ff -m ‘说明信息’ dev 强制禁用快速合并，会做一次新的commit
    • bug分支：每个bug都可以创建一个临时分支 修复后合并分支 然后把临时分支删除
    • git stash 暂时保存工作现场
    • Git stash list 列出保存的工作现场
    • git stash pop 回复工作现场

      github

    • README 介绍项目的
    • .gitignore 空时忽略文件
    • ssh-keygen -t rsa -C ‘邮箱地址’
    • git clone ssh地址 克隆代码
    • 如果克隆出错
      • eval “$(ssh-agent -s)” ssh-add 执行这两句话
    • git push origin 分支名称 推送本地分支到远程分支
    • git branch –set-upstream-to=origin/远程分支名 本地分支名 设置本地分支跟踪远程分支
    • 跟踪后 git push就可以把本地版本推送到远程分支
    • git pull orgin 分支名称 拉取远程分支代码

      发送邮件

    • 发邮件是django发送到smtp上面 然后smtp服务器发送给目的邮箱。且django发送到smtp服务器时间是不确定的，会让程序卡在这

        celery

    • pip install celery
    • 如果需要把celery 形成工程目录
    • celery -A celery_tasks.tasks worker -l info 启动任务者
      • 启动的时候 要在相同的目录启动项目和celery的worker
      • 如果在win10 要安装 eventlet 然后 celery -A celery_tasks.tasks worker -l info -P eventlet
      • windows在使用celery的时候，会存在很多坑。比如celery_task目录下的文件不能为tasks.py文件，必须是celery.py文件。同时在启动celery的work服务时，需要下载一个依赖包eventlet。启动时必须使用如下的命令（对应的目录下执行）：celery worker -A celery_task -l info -P eventlet
    • 如果需要接受worker的结果 要用到backend存放结果数据
      • 返回的结果是异步结果的对象 通过 返回值.get()能获取celery异步执行的结果（会变成同步行为 会有执行结果后才继续执行下面） 添加timeout 超过时间 立刻返回

      悲观锁和乐观锁

    • 悲观锁 上来就加锁 自己用了 不让别人用
    • 乐观锁 觉得别人不会修改 先尝试修改 然后提交时判断和原来是否一样 一样就提交 不一样就失败
